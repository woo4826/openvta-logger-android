# OpenVTA Logger Data Format

## Storage Layout

Default app-private directory:

```text
files/vta/sessions/
```

Typical files per session:

```text
VTA<ddMMyyyy>_<HHmmss>_<driverId>.Vta
VTA<ddMMyyyy>_<HHmmss>_<driverId>.meta
VTA<ddMMyyyy>_<HHmmss>_<driverId>.Zip
```

Example:

```text
VTA18062026_182548_CC00.Vta
VTA18062026_182548_CC00.meta
VTA18062026_182548_CC00.Zip
```

## VTA File Purpose

The `.Vta` file is the compatibility surface for existing VTA-style analysis workflows. OpenVTA keeps legacy-style records while adding metadata and enhanced-position records.

## Record Types

### `$` Raw GPS Record

`$` rows are raw Android GPS fixes.

Use these as the source of truth for measured location.

Typical fields include:

- time
- latitude
- longitude
- altitude
- speed
- bearing
- accuracy
- provider
- elapsed realtime
- satellite count when available

### `@` Enhanced GPS-Like Estimate Record

`@` rows are derived estimates generated by the selected IMU/GPS enhancement preset.

Use these for visualization or analysis experiments, not as hardware GPS measurements.

Typical extra fields include:

- source
- confidence
- selected preset
- raw GPS interval used for derivation
- timestamp used for the estimated point

### `#` Sensor Record

`#` rows are phone sensor samples.

Typical sensor data includes:

- accelerometer
- gyroscope
- orientation/heading-related values
- rotation vector
- sensor timestamp
- sensor accuracy

## Header Records

The file header includes OpenVTA/VTA compatibility metadata, including format version and selected IMU preset.

Important header concept:

```text
%% ImuPresetId
```

This records the preset used for that session.

## End Record

Closed sessions should contain:

```text
%% End
```

If this is missing, the session may have been interrupted or the app/service was killed before a clean stop.

## Raw Versus Enhanced Interpretation

Never mix `$` and `@` as if both are measured GPS.

Recommended analysis workflow:

1. Use `$` rows for ground-truth GPS-based analysis.
2. Use `@` rows for visualization or comparison.
3. Use `#` rows to inspect sensor context.
4. Compare raw and enhanced traces separately.

## IMU/GPS Presets And Export Behavior

| Preset | Export behavior | Best use |
| --- | --- | --- |
| Raw GPS | Mostly `$`, minimal or no `@` | baseline, compatibility, measurement-first analysis |
| Linear 5Hz | `$` plus linearly interpolated `@` | simple smoother map path |
| Hermite 10Hz | `$` plus curved `@` estimates | smoother visualization |
| IMU heading 10Hz | `$`, `#`, and heading-influenced `@` | fixed phone mount, heading experiments |
| Bounded IMU dead reckoning 10Hz | `$`, `#`, and bounded IMU-based `@` | short-gap IMU experiment |

## ZIP Format

ZIP creation keeps the session export simple:

- one `.Vta` file per `.Zip`
- entry name should match the `.Vta` basename
- ZIP is generated on demand from the Sessions screen or before upload

## OpenVTA Live Upload Boundary

OpenVTA Live uploads the same VTA-compatible session bytes through the backend
API after explicit user pairing. The Android app must not write Live API
credentials, WSS/MQTT credentials, six-digit pairing codes, object-storage
keys, or signed URLs into `.Vta` or `.Zip` exports.

The backend stores VTA bytes in private object storage and metadata in
PostgreSQL. Android treats Live upload state as transport state, not as part of
the local export format.

## Privacy

Treat `.Vta` and `.Zip` as sensitive. They may include:

- precise location
- speed
- heading/bearing
- altitude
- timestamps
- driver/session identifier
- phone sensor data
- provider and accuracy metadata

Do not attach real user sessions to public issues or commits unless they are deliberately anonymized.

## Compatibility Notes

- Legacy consumers should continue reading `$` and `#`.
- New consumers should add support for `@` but keep it separate.
- If a downstream tool does not understand `@`, it should ignore those rows rather than treating them as raw GPS.
